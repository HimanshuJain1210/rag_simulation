📄 Step 1 — Document Chunking Retrieval
Raw documents are split into smaller, overlapping chunks. Each chunk is small enough to fit in an LLM context window but large enough to be semantically meaningful.
💡 Why chunk? LLMs have token limits. Chunking lets you index thousands of pages and only pull the relevant slice.
🔢 Step 2 — Vector Embedding Retrieval
Each chunk is passed through an embedding model (e.g. text-embedding-ada-002). It converts text into a list of numbers — a vector — that captures semantic meaning.

Resulting embedding vector (first 32 dims)
💡 Vectors encode meaning, not just keywords. "Car" and "Automobile" land near each other in vector space.
🗄️ Step 3 — Vector Database Retrieval
All chunk vectors are stored in a Vector DB (e.g. Pinecone, Weaviate, FAISS). This enables lightning-fast nearest-neighbour search at scale.

📡 Step 4 — Similarity Search Retrieval
Your query is also embedded. Its vector is compared against all stored vectors using cosine similarity. The top-K most similar chunks are retrieved.

🧩 Step 5 — Prompt Augmentation Augment
The retrieved chunks are injected into the prompt as context. The LLM now has both your question and the relevant knowledge — no fine-tuning needed.

💡 This is the "A" in RAG. The model isn't memorising your data — it's reading it fresh every time, like an open-book exam.
✨ Step 6 — LLM Generation Generate
The LLM reads the augmented prompt and generates a grounded answer, citing only what's in the provided context — reducing hallucination.
